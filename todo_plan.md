## Найденные расхождения и недочёты

### 1. `cmd/chezget/main.go` — 0% покрытия тестами

`main()` не покрыт юнит-тестами. Хотя это тонкая обёртка и общий coverage
проекта (89.4%) удовлетворяет требованию ≥ 60%, а критичные пакеты
превышают 80%, нулевое покрытие точки входа снижает общую картину.

**Доработка:** вынести логику `main()` в тестируемую функцию
(например `run(args []string, stdout, stderr io.Writer) int` в `cmd/chezget`
или `internal/app`) и добавить тесты на:
- `--version` (выводит `chezget <version>` и выходит с кодом 0)
- `--config <путь>` (передаёт путь в app)
- поведение без аргументов / с неизвестным флагом

**Приоритет:** средний.
**Ожидаемый эффект:** coverage `cmd/chezget` поднимется с 0% до ≥ 80%.

### 2. `app.hintPath()` — непокрытая ветка ошибки (66.7%)

Ветвь `ResolvePath()` → ошибка (фоллбэк на строку
`"$XDG_CONFIG_HOME/chezget/config.ini"`) не протестирована.

**Доработка:** добавить тест, в котором `XDG_CONFIG_HOME` не задан, а
`HOME`/`USERPROFILE` недоступен, чтобы `ResolvePath` возвращала ошибку, и
проверить, что `hintPath()` возвращает резервную строку.

**Приоритет:** низкий.
**Ожидаемый эффект:** coverage `hintPath` → 100%, `app` → ~96%.

### 3. Заголовки лицензии MIT в файлах исходного кода

Дизайн-документ требует: «Исходники проекта распространяются по лицензии
MIT и оформлены в соответствии с требованиями лицензии.» Файл `LICENSE`
в корне удовлетворяет формальным требованиям MIT (уведомление в «substantial
portions»), однако ни один `.go`-файл не содержит шапки с указанием
авторских прав и ссылкой на LICENSE. Добавление заголовков — распространённая
конвенция, усиливающая соответствие «оформлены в соответствии».

**Доработка:** добавить в каждый `.go`-файл шапку вида:

```go
// SPDX-License-Identifier: MIT
// Copyright (c) 2026 Alex 'Ript' Malyshev
```

(или аналогичную, принятую в проекте).

**Приоритет:** низкий (по усмотрению).
**Ожидаемый эффект:** однозначное оформление источников под MIT.

### 4. Bare `go build` из корня проекта не работает

`designdoc.md` формулирует: «Приложение должно полностью собираться
инструкцией `go build`.» Запуск `go build` без аргументов из корня
завершается с `no Go files in ...`, поскольку entrypoint находится в
`cmd/chezget/` (что соответствует современным рекомендациям Go).
Сборка работает через `go build ./cmd/chezget` или `make build`.

Это не баг, а следствие стандартной структуры проекта, однако возможна
строгая трактовка требования. Если требуется, чтобы bare `go build`
работал из корня, есть два варианта:

- **Вариант A (предпочтительный):** оставить как есть, уточнив формулировку
  в `designdoc.md` / `README.md` / `AGENTS.md`, что инструкция сборки —
  `go build ./cmd/chezget` (или `make build`).
- **Вариант B:** добавить корневой `go.work` — не решает проблему (bare
  `go build` всё равно не собирает main без указания пакета).
- **Вариант C:** вынести `main.go` в корень — противоречит «современным
  рекомендациям», не рекомендуется.

**Доработка:** согласовать формулировку; при выборе варианта A —
явно документировать команду сборки.

**Приоритет:** низкий (интерпретация требования).
**Ожидаемый эффект:** однозначность инструкции сборки.

### 5. `go.mod`: директива `go 1.26.0`

В `go.mod` указано `go 1.26.0` (с патчем). Это допустимо с Go 1.21+, но
общепринятая практика — указывать `go 1.26` без патча, если не требуется
фиксация конкретного минорного релиза.

**Доработка:** по желанию заменить `go 1.26.0` → `go 1.26`.

**Приоритет:** косметический.
